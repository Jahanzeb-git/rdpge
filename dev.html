RDPGE Runtime Dynamic & Probabilistic Graph Execution

RDPGE is a production-grade agentic framework designed to solve the "Contextual Myopia" problem in classic ReAct loops. By treating execution as a dynamic graph rather than a flat list, it enables agents to maintain massive context windows through selective blurring and active recall.

Python-First Philosophy

RDPGE replaces JSON tool calls (which are prone to formatting errors) with Python Code Generation. Instead of hoping the LLM formats JSON correctly, we let it write Python code that we sandboxed-execute.

Installation

pip install rdpge

Requirements: Python 3.10+

Quick Start

Here is a complete example of creating an agent that can read files and perform reasoning.

import asyncio
from rdpge import Agent, tool
from rdpge.llm import OpenAIAdapter, LLMConfig

# 1. Define Tools
@tool("Read file contents")
def read_file(path: str) -> str:
    """Reads a file from the disk."""
    try:
        with open(path, 'r') as f:
            return f.read()
    except FileNotFoundError:
        return "File not found."

# 2. Configure LLM
llm = OpenAIAdapter(LLMConfig(
    model="gpt-4o",
    api_key="your-api-key"
))

# 3. Initialize Agent
agent = Agent(
    llm=llm,
    tools=[read_file],
    instructions="You are a helpful coding assistant."
)

# 4. Run Agent
async def main():
    print("Running agent...")
    result = await agent.run("Read 'README.md' and summarize it.")
    
    print(f"Status: {result.status}")
    print(f"Answer: {result.reason}")

if __name__ == "__main__":
    asyncio.run(main())

Agent Configuration

The Agent class is highly configurable. You can load instructions from files and strict type checking.

Loading Instructions from File

Instead of hardcoding big prompt strings, you can load them from a markdown file. This is best practice for production agents.

agent = Agent(
    llm=llm,
    tools=[...],
    instructions_file="./prompts/coder_instructions.md" 
)

This allows you to version control your implementation prompts separately from your code.

Multi-Turn Execution

RDPGE supports long-running sessions where the agent remembers previous interactions. The context is automatically managed.

# First turn: Starts a new session
result1 = await agent.run("Scan the codebase for auth issues.")
print(f"Session ID: {result1.session_id}")

# Second turn: Continues the SAME session
# The agent remembers what it scanned in the first turn!
result2 = await agent.run("Fix the issue you found in auth.py")

Session Management

  • agent.new_session(): Wipes memory to start fresh.
  • agent.session_id: Access the current active session ID.

Defining Tools

Tools are the bridge between the LLM and the outside world. RDPGE provides a flexible way to define them.

The @tool Decorator

The simplest way. It automatically uses type hints and docstrings for the LLM schema.

@tool()
def calculate(a: int, b: int) -> int:
    """Adds two numbers."""
    return a + b

BaseTool Class

For tools requiring state (like database connections), subclass BaseTool. This gives you setup() and teardown() lifecycle methods.

from rdpge import BaseTool

class DatabaseTool(BaseTool):
    name = "query_db"
    description = "Execute SQL query"

    async def setup(self):
        self.conn = await connect_to_db()

    async def execute(self, query: str) -> str:
        return await self.conn.query(query)

    async def teardown(self):
        await self.conn.close()

Signals & Control

RDPGE provides built-in "Signal Tools" to control the execution flow.

Signal Description
complete Task finished successfully. Ends the loop.
ask_user Agent pauses execution to ask the user a question. Returns status awaiting_input.
surrender Agent gives up. Ends the loop with failure status.
restore_context Restores the "blurred" context of a previous task. Used for memory recall.

Observability & Hooks

RDPGE is designed to be transparent. You can hook into every event of the agent's lifecycle.

Event Hooks

@agent.on("step_start")
async def on_step(data):
    # data = { "step": 1, "session_id": "...", "active_node": "..." }
    print(f"Starting Step {data['step']}")

@agent.on("tool_called")
async def on_tool(data):
    # data = { "tool": "read_file", "args": {...}, "output_length": 120 }
    print(f"Tool Used: {data['tool']}")

@agent.on("error")
async def on_error(data):
    print(f"Error at step {data['step']}: {data['error']}")

Exporting Execution Graphs

You can export the entire execution graph (nodes, edges, code, output) to a JSON file for analysis or visualization.

result = await agent.run(...)
path = result.export_graph("./traces/session_dump.json")
print(f"Graph dumped to: {path}")

Persistence

By default, RDPGE uses an in-memory store. For production apps, you should persist state to a database (Redis, Postgres, etc.) so sessions survive restarts.

Implementing a Custom Store

Implement the SessionStore protocol:

from rdpge.storage.base import SessionStore
import redis.asyncio as redis
import json

class RedisStore(SessionStore):
    def __init__(self, url):
        self.r = redis.from_url(url)

    async def save(self, session_id: str, state: dict):
        # State is a simple dict, easy to serialize
        await self.r.set(session_id, json.dumps(state))

    async def load(self, session_id: str) -> dict:
        data = await self.r.get(session_id)
        return json.loads(data) if data else None

    async def delete(self, session_id: str):
        await self.r.delete(session_id)

    async def list_sessions(self) -> list[str]:
        keys = await self.r.keys("*")
        return [k.decode() for k in keys]

# Usage
agent = Agent(..., store=RedisStore("redis://localhost"))

API Reference: Agent

class Agent(llm, tools=None, instructions="", instructions_file=None, max_steps=25, store=None)

run(request: str) -> AgentResult

Executes the agent loop. If called multiple times with the same instance, it continues the same session.

new_session()

Resets the internal state to start a fresh conversation context (new session ID).

abort(reason: str)

Signals the agent to stop at the next step. Useful for UI "Stop" buttons.

load_session(session_id: str) -> bool

Loads a past session from the Store. Returns True if found.

API Reference: AgentResult

@dataclass AgentResult
Attribute Type Description

status str 'completed', 'surrendered', 'aborted', 'awaiting_input', 'max_steps'
